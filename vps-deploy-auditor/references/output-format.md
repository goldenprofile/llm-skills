# Формат вывода

Два режима — два формата. В режиме генерации выдаёшь шаблоны, в режиме аудита — отчёт.

## Режим генерации

Когда пользователь просит конфиги под новый деплой:

- Уточни минимум контекста (тип приложения, домен, socket/порт, есть ли postgres/redis) — см.
  раздел «Контекст» в SKILL.md. Не выпытывай лишнее, бери разумные дефолты под профиль.
- Выдай готовые к правке шаблоны: unit-файл, server-блок nginx (если HTTP), cron-бэкап,
  настройку redis/postgres — с плейсхолдерами (`example.com`, `myapp`, `appuser`) и краткими
  комментариями, ПОЧЕМУ так.
- В конце — список команд установки по шагам (создать пользователя, `daemon-reload`,
  `enable --now`, `certbot`, `ufw`) и чеклист «перед первым стартом».
- Не придумывай несуществующие директивы/флаги — только корректные актуальные практики.

## Режим аудита

Сначала вердикт, потом находки по убыванию риска, потом правки.

```markdown
## Аудит деплоя — вердикт

**Готов к проду:** ❌ НЕТ / ⚠️ С ОГОВОРКАМИ / ✅ ДА
**Стек:** Django 5.x · gunicorn (socket) · nginx+TLS · postgres 16 · redis (cache) · systemd · ufw
**Проверено:** unit-файл, server-блок nginx, pg_hba, redis.conf, бэкап-cron, ufw status, права .env

Одной строкой: что роняет деплой / уязвимо / переживёт ли рестарт.

---

## Находки

### 🔴 CRITICAL — <краткое имя>
- **Где:** `/etc/postgresql/16/main/pg_hba.conf` строка N / `myapp.service` / `redis.conf`
- **Что не так:** напр. «нет бэкапа postgres — при потере диска данные не восстановить».
- **Почему критично у тебя:** <одиночный VPS, соло-эксплуатация, нет реплики/off-site>
- **Как исправить:** <конкретный конфиг/команда: pg_dump-cron + off-site + проверка restore>

### 🟠 HIGH — <…>
- **Где / Что не так / Почему / Как исправить** — тот же формат.

### 🟡 MEDIUM — <…>
### ⚪ LOW — <…>

---

## Предлагаемые правки

Для каждой принятой находки — конкретный конфиг/команда (строки unit-файла, блок nginx,
строки redis.conf/postgresql.conf, cron-запись, команды ufw). С пометкой о порядке применения
(например: «daemon-reload → restart», «certbot после DNS», «ufw enable последним, не отрезав SSH»).

## Чеклист перед/после деплоя
- [ ] Сервис под не-root пользователем, Restart=always, enable-нут (переживает ребут)
- [ ] TimeoutStopSec ≥ graceful-timeout приложения
- [ ] Секреты в EnvironmentFile (chmod 600), не в репо/в unit-строке
- [ ] HTTP за nginx: TLS (certbot) + редирект 80→443 + auto-renew проверен
- [ ] nginx: статика/медиа через alias, gzip, security-заголовки, server_tokens off
- [ ] postgres: отдельный пользователь/БД, pg_hba без trust, listen только localhost
- [ ] postgres: pg_dump по cron + проверенный restore + off-site копия
- [ ] redis: bind 127.0.0.1; maxmemory+policy (cache) или AOF (broker/FSM)
- [ ] ufw: открыты только 22/80/443; БД/redis не наружу
- [ ] max_connections согласован с числом воркеров
- [ ] Миграции применяются безопасно (migration-safety-auditor)
```

## Принципы вывода

- **Вердикт первым** — «можно ли катить / безопасно ли», потом детали.
- **Конкретика:** файл, строка, конкретная директива и команда исправления, а не общие слова.
- **Не раздувай риск.** Для pet-проекта на 10 пользователей отсутствие fail2ban — LOW, а
  отсутствие бэкапа БД с реальными данными — всё равно CRITICAL. Привязывай уровень к ценности
  данных и доступности извне.
- **Безопасность периметра — выше косметики.** root/секреты-в-репо/нет-бэкапа/БД-наружу важнее,
  чем gzip и заголовки.
- **Правки — по согласованию:** сначала отчёт, потом меняем то, что владелец взял в работу.
  Деструктивные шаги (ufw enable, restart postgres, смена pg_hba) — с предупреждением о порядке,
  чтобы не отрезать себе SSH/доступ.
