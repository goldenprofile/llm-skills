# Ratchet Loop — маркеры транскрипта и память

## Маркеры транскрипта

В дефолтном in-session режиме маркеры — самодисциплина и человекочитаемые чекпоинты
(хост-оценщика нет). Под опциональной обёрткой `/goal` те же маркеры становятся
контрактом для evaluator'а, читающего транскрипт.

| Маркер | Когда |
|---|---|
| `RL_CONTRACT` | этап 1, зафиксированный контракт |
| `RL_EVAL_LOCKED` | этап 2, файлы frozen-set + хеши |
| `RL_BASELINE: <metric>` | этап 3 |
| `RL_KEEP: <metric>` / `RL_REVERT` / `RL_CRASH` | по разу на попытку (решение храповика) |
| `RL_VERIFY` | независимый проход (чекпоинт/финал) |
| `MEMORY_SAVED: <name\|none>` | перед завершением |
| `RL_HALT` | eval-tamper / 3-strike / verify-fail / чекпоинт / сайд-эффект — **возврат управления** |
| `RL_RUN_COMPLETE` | финал, после зелёного `RL_VERIFY` |

## Память (Windows)

Memory-каталог бери из того, что харнесс показывает в начале сессии (путь зависит от
проекта — обычно `…\.claude\projects\<slug>\memory`). Формат — твой стандартный: frontmatter
`name` / `description` / `metadata.type` (`reference`/`project`/`feedback`), тело,
линки `[[name]]`, строка-указатель в `MEMORY.md`. На финале всегда пиши
`ratchet_<artifact-slug>.md`: что за артефакт, метрика и направление, epsilon, какие
гипотезы сработали/провалились, лучший дифф — чтобы следующий прогон стартовал умнее
и не повторял мёртвые ветки.
