# Формат вывода аудита

Сначала вердикт, потом находки по убыванию риска, потом план/правки. Конкретика: пакет,
версия, advisory-ID, прод/dev, fix-версия — а не общие слова.

```markdown
## Аудит зависимостей — вердикт

**Безопасно деплоить:** ❌ НЕТ / ⚠️ С ОГОВОРКАМИ / ✅ ДА
**Стек:** uv · pyproject + uv.lock (с хешами) · prod/dev разделены · Python 3.12
**Проверено:** 47 пакетов (12 прямых), pip-audit + safety, lockfile, outdated, прямые имена

Одной строкой: что уязвимо/невоспроизводимо или почему всё в порядке.

---

## Находки

### 🔴 CRITICAL — <краткое имя>
- **Пакет:** `requests==2.25.0` (прод, прямая) · advisory `GHSA-xxxx` (CVE-2023-xxxxx)
- **Что не так:** <CVE с эксплойтом в прод-пути / typosquat / скомпрометированный пакет>
- **Почему опасно у тебя:** <прод vs dev, на пути выполнения, есть ли воркэраунд>
- **Fix:** обновить до `>=2.31.0` (fix-версия из pip-audit) → `uv lock --upgrade-package requests`

### 🟠 HIGH — <…>
- **Пакет / Что не так / Почему / Fix** — тот же формат.
  Примеры: CVE в прод-зависимости; нет lockfile при деплое; установка из непиннутого диапазона.

### 🟡 MEDIUM — <…>
  Примеры: CVE только в dev/test; устаревший на мажор без CVE; prod/dev не разделены; lock без хешей.

### ⚪ LOW — <…>
  Примеры: patch/minor-отставание, стиль пиннинга, нет проверки лицензий.

---

## Уязвимости (сводка скана)

| Пакет | Версия | Advisory | Severity | Fix | Прод/Dev |
|-------|--------|----------|----------|-----|----------|
| requests | 2.25.0 | CVE-2023-… | HIGH | 2.31.0 | prod |
| <…> | | | | «No known fix» → митигейшн | dev |

## План безопасного обновления

Для каждой принятой находки — конкретный шаг, по одному/батчами, с пометкой риска semver:
1. `requests` 2.25→2.31 (minor, low) — батч с прочими patch/minor.
2. `pydantic` 1→2 (MAJOR, breaking) — отдельная ветка, migration guide, тесты.
После каждого шага: `pytest` → `pip-audit` (ре-скан) → коммит lockfile.

## Чеклист перед деплоем
- [ ] pip-audit (и safety) — нет CRITICAL/HIGH в прод-пути, либо есть обоснованный митигейшн
- [ ] Lockfile существует, с хешами, закоммичен
- [ ] Установка воспроизводима (uv sync --frozen / --require-hashes / poetry install)
- [ ] Прямые прод-зависимости запинены осознанно, prod/dev разделены
- [ ] Нет заброшенных прод-пакетов с открытой уязвимостью без плана замены
- [ ] Прямые имена пакетов сверены (нет typosquat)
- [ ] После апгрейдов тесты зелёные, старый lockfile сохранён для отката
```

## Принципы вывода

- **Вердикт первым** — «можно ли деплоить», потом детали.
- **Контекст определяет риск.** CVE в dev-линтере ≠ CVE в прод-фреймворке. Прод-путь и
  наличие fix-версии важнее голого CVSS. Не раздувай и не занижай.
- **Точность инструментов.** Указывай реальные advisory-ID, fix-версии и команды менеджера
  пользователя (uv/poetry/pip), а не абстрактные «обновите пакет».
- **«No known fix» — это митигейшн, не апгрейд.** Фиксируй принятый риск явно (что, почему,
  до какого условия), а не молчаливый `--ignore-vuln`.
- **Правки — по согласованию.** Сначала отчёт; обновляй и пиннингуй то, что владелец взял в
  работу. Апгрейды — маленькими протестированными шагами, не «всё до latest» одним махом.
```
