# Безопасное обновление зависимостей

Принцип: обновление — это изменение кода, даже если ты не трогал ни строки. Любой апгрейд
должен быть проверяем (тесты) и обратим (старый lockfile). Лучший способ не получить инцидент —
**маленькие, изолированные, протестированные** шаги.

## Semver — как читать версии

`MAJOR.MINOR.PATCH`:

- **PATCH** (`5.0.1 → 5.0.2`) — только багфиксы/безопасность, обратно совместимо. Низкий риск.
- **MINOR** (`5.0 → 5.1`) — новые фичи, обратная совместимость **обещана**, но бывают
  deprecations и поведенческие изменения. Средний риск — читай changelog.
- **MAJOR** (`5.x → 6.0`) — **breaking changes** ожидаемы. Высокий риск, отдельный апгрейд.

Оговорки: (1) semver — это соглашение, не гарантия; пакеты его нарушают. (2) До `1.0.0`
(`0.x.y`) правила слабее — даже minor может ломать. (3) Безопасность может прилететь в старой
ветке (`5.0.x`) — не всегда нужно прыгать на свежий мажор, чтобы закрыть CVE.

## Перед апгрейдом

1. **Чистая отправная точка**: рабочее дерево без незакоммиченных правок, тесты зелёные сейчас.
2. **Зафиксируй откат**: текущий lockfile в git — это и есть точка возврата.
3. **Узнай, что меняется**: changelog/release notes пакета (GitHub Releases, `CHANGELOG.md`,
   страница PyPI). Ищи разделы *Breaking changes*, *Deprecations*, *Removed*, *Migration*.
   Для крупных (Django, Pydantic, SQLAlchemy) — официальный гайд миграции мажора.
4. Оцени **диапазон скачка**: patch/minor/major и сколько версий перепрыгиваешь.

## Стратегия: по одному vs батчами

- **По одному пакету** — для мажорных апгрейдов, для всего, что близко к прод-логике (web-фреймворк,
  ORM, драйвер БД, валидация), и для закрытия конкретного CVE. Легко локализовать поломку и откатить.
- **Батчами** — приемлемо для пачки patch/minor-апдейтов «тихих» пакетов (утилиты, dev-tools).
  Если батч ломает тесты — разбивай пополам, пока не найдёшь виновника.

Не делай «обнови всё до latest» одним коммитом на боевом проекте: при поломке непонятно, что
именно сломалось, и откат теряет все полезные апдейты.

## Рабочий процесс (на ветке)

```bash
# 1. отдельная ветка
git switch -c deps/upgrade-django-5.1

# 2. поднять ОДИН пакет (пример для каждого менеджера)
uv lock --upgrade-package django && uv sync          # uv
poetry update django                                  # poetry
pip-compile --upgrade-package django requirements.in && pip-sync requirements.txt   # pip-tools

# 3. посмотреть, что реально изменилось в графе
git diff uv.lock          # / poetry.lock / requirements.txt — какие транзитивные тоже поехали

# 4. прогнать тесты (см. навык test-coverage-auditor)
pytest -q                  # или make test

# 5. ре-скан уязвимостей: убедиться, что CVE закрыт и не приехал новый
pip-audit

# 6. ручная проверка затронутого поведения (если апгрейд рискованный)
```

После зелёных тестов — коммит lockfile+манифеста, PR (ревью — `techlead-ai`).

## После апгрейда — что проверять

- **Тесты** зелёные. Если покрытие слабое — апгрейд рискованнее; усиль тесты затронутого кода
  (см. `test-coverage-auditor`) **до** того, как доверять «тесты прошли».
- **Deprecation warnings** в выводе тестов/запуска — будущие поломки; не игнорируй.
- **Запуск приложения** (smoke): стартует, основные эндпоинты/команды работают.
- **Ре-скан** `pip-audit`: целевой CVE закрыт, новых не появилось.
- **Изменения в lockfile**: какие транзитивные подтянулись — нет ли среди них уязвимых/мажорных.

## Откат

- Самый надёжный — `git checkout <старый lockfile> manifest` и пере-`sync`/`install`:
  `git checkout HEAD~1 -- uv.lock pyproject.toml && uv sync --frozen`.
- Поэтому lockfile обязан быть в git **до** апгрейда — иначе откатываться не к чему.
- Если апгрейд уже на проде и ломает — откат на предыдущий известный-хороший lock и редеплой
  быстрее, чем отладка нового пакета в проде.

## Частые источники breaking changes (Python-бэкенд)

- **Django** мажор: удалённые/перемещённые API, изменения в ORM/middleware, новые дефолты;
  всегда есть release notes с разделом deprecations по версиям. Связка с миграциями БД —
  см. навык `migration-safety-auditor`.
- **Pydantic 1 → 2**: переписан валидатор, изменены API (`@validator`→`@field_validator`,
  `.dict()`→`.model_dump()`); крупная миграция, только отдельным шагом.
- **SQLAlchemy 1.4 → 2.0**: новый стиль запросов/сессий.
- **httpx/urllib3/requests**: смена дефолтов таймаутов, проверка TLS, поведение редиректов.
- Драйверы БД (`psycopg2`→`psycopg3`), `redis-py` мажоры — меняют API подключения.

Для таких — отдельная ветка, чтение migration guide, точечные тесты, не в общем батче.

## Поддержание в актуальном состоянии (профилактика)

- `pip-audit` шагом CI (`make sec` из `harness-engineering`) — ловит CVE при каждом PR.
- Регулярный плановый прогон `*-outdated` (раз в спринт), а не «раз в год большим скачком» —
  маленькое отставание дешевле и безопаснее обновлять.
- Dependabot/Renovate для PR-ов с апдейтами — но мерж только после тех же тестов и ре-скана.
