# Скан уязвимостей и проблемных пакетов

Два основных инструмента: `pip-audit` (PyPA, сверяет с базой PyPI Advisory + OSV) и
`safety` (база Safety DB). Их выводы частично пересекаются, но не идентичны — для серьёзного
аудита прогоняй оба. Дополнительно — проверка устаревших, заброшенных пакетов и typosquatting.

## pip-audit

Установка: `pip install pip-audit` (или `uv tool install pip-audit`, чтобы не пачкать venv).

```bash
# Скан текущего окружения (резолвит установленные пакеты)
pip-audit

# Скан по файлу требований без установки в окружение
pip-audit -r requirements.txt

# Скан проекта по pyproject.toml (резолвит зависимости проекта)
pip-audit .

# Машиночитаемый вывод (для CI / парсинга)
pip-audit -f json -o pip-audit.json

# Markdown-таблица (удобно вставлять в отчёт/PR)
pip-audit -f markdown

# Показывать и описание уязвимости
pip-audit --desc

# Пытаться автоматически подобрать безопасные версии (только показывает/чинит requirements)
pip-audit -r requirements.txt --fix --dry-run

# Игнорировать конкретный advisory (с обоснованием в отчёте, не молча)
pip-audit --ignore-vuln GHSA-xxxx-xxxx-xxxx
```

Полезные флаги: `-l/--local` (только локальные, без PyPI-lookup версий), `--require-hashes`
(вместе с `-r` для проверки хешей), `-s osv|pypi` (выбор источника advisory), `--strict`
(падать, если у пакета нельзя извлечь зависимости).

**Чтение вывода.** Строка содержит: имя пакета, текущую версию, ID advisory
(`GHSA-…`/`PYSEC-…`/`CVE-…`) и колонку **Fix Versions**. Логика разбора:

- Есть Fix Version → планируй апгрейд до неё (см. `upgrades.md`).
- Fix Version пустая («No known fix») → апдейтом не закрыть; нужен митигейшн (ниже).
- Пакет — транзитивный (его нет в твоих манифестах) → чинится бампом родителя или
  временным пиннингом в constraints; см. раздел про транзитивные.

`pip-audit` завершается **ненулевым кодом**, если найдены уязвимости, — это и делает его
пригодным для CI-гейта (шаг `make sec` в `harness-engineering`).

## safety

Современный CLI — `safety scan` (старая команда `safety check` устарела, многие фичи требуют
бесплатной регистрации/`safety auth`). Если регистрация нежелательна — основным инструментом
держи `pip-audit`, а `safety` используй как второе мнение.

```bash
pip install safety
safety scan                          # скан проекта/окружения
safety scan --output json            # машиночитаемо
# Легаси, но работает офлайн по файлу:
safety check -r requirements.txt --output text
```

`safety` группирует по severity (CRITICAL/HIGH/MEDIUM/LOW на основе CVSS) — используй эти
оценки как ориентир для уровня риска в отчёте.

## Как трактовать severity → уровень риска отчёта

Severity advisory (CVSS) — это базовая оценка, но **контекст важнее**:

- CVE в **прод**-зависимости на пути выполнения, есть fix → **CRITICAL/HIGH**.
- Тот же CVE только в **dev/test**-зависимости (линтер, тестраннер) → понижай до **MEDIUM**:
  он не в проде. Но если dev-пакет исполняется в CI с доступом к секретам — не недооценивай.
- CVE требует условий, которых у проекта нет (напр. отключённая фича, недостижимый код-путь) →
  отметь в отчёте, понизь риск, но не игнорируй молча.
- «No known fix» → риск не снимается; решай митигейшном, а не апгрейдом.

## Что делать с найденным CVE (порядок предпочтения)

1. **Апгрейд до fix-версии** — основной путь. Для прямой зависимости — бамп в манифесте; для
   транзитивной — бамп родителя или пиннинг (ниже). Затем тесты (см. `upgrades.md`).
2. **Пиннинг транзитивной зависимости** до безопасной версии, если родитель ещё не выпустил
   обновление: добавить пакет как прямую зависимость с `>=fixed` или через constraints-файл.
3. **Временный митигейшн**, если fix нет/несовместим:
   - отключить/обойти уязвимый код-путь, ограничить вход;
   - сетевые/конфиг-ограничения (не принимать недоверенный ввод в уязвимую функцию);
   - зафиксировать advisory как принятый риск (`--ignore-vuln <ID>`) **с записью в отчёт** —
     что, почему, до какого срока/условия.
4. **Замена пакета**, если он заброшен и патча не будет (см. ниже).

## Транзитивные зависимости

Уязвимость часто сидит в пакете, которого нет в твоих манифестах. Найти путь:

```bash
uv tree --invert --package urllib3     # кто тянет urllib3 (uv)
pipdeptree -r -p urllib3               # обратное дерево (pip, нужен pipdeptree)
poetry show urllib3                    # poetry: показывает зависимости и зависимых
```

Починка: (а) обновить родителя, который уже требует безопасную версию; (б) если родитель
отстаёт — закрепить безопасную версию транзитивного пакета через **constraints**
(`pip install -c constraints.txt`, `requirements`-constraints) или добавить его прямой
зависимостью с `>=`; в poetry/uv — добавить override/прямую зависимость. Зафиксируй причину
комментарием, чтобы пиннинг не «завис» навечно.

## Устаревшие пакеты

```bash
pip list --outdated                    # Version vs Latest для установленных
uv pip list --outdated                 # то же в uv-окружении
uv tree --outdated                     # дерево с пометкой устаревших (uv)
poetry show --outdated                 # poetry
```

Колонки: текущая, последняя совместимая, последняя вообще. Отставание на patch/minor — LOW;
отставание на **мажор** — MEDIUM (есть breaking changes и накопленный риск, см. `upgrades.md`).

## Заброшенные / непопулярные пакеты

Признаки риска: нет релизов 1–2+ года, архивный/несуществующий репозиторий, один мейнтейнер
без активности, мало загрузок и звёзд при критичной роли в проде, открытые security-issue без
ответа. Проверка вручную: страница PyPI (дата последнего релиза), `pip show <pkg>` (Home-page,
Author), репозиторий (последний коммит, открытые issues), статистика загрузок (pypistats.org).
Заброшенный прод-пакет с открытой уязвимостью без перспективы патча → **HIGH** + рекомендация
замены на поддерживаемый аналог.

## Typosquatting и supply-chain (базовая осведомлённость)

Атака — пакет с именем, похожим на популярный (`python-requests` вместо `requests`, `djnago`
вместо `django`, `urlib3`, `python-sqlite`). Проверка:

- Сверяй каждое **прямое** имя из манифеста с реальным именем на PyPI и в документации проекта.
- Подозрительно: пакет, появившийся недавно, с мало загрузками, имитирующий популярное имя.
- Закрепляй версии и используй **хеши** (`--require-hashes`) — защищает от подмены артефакта.
- Ставь только из доверенного индекса; осторожно с приватными индексами и
  dependency-confusion (внутреннее имя, перехваченное публичным PyPI).

Полная защита supply-chain — за рамками навыка (SBOM, подписи, OIDC-publishing), но typosquat
и невоспроизводимую установку аудит обязан отлавливать.
