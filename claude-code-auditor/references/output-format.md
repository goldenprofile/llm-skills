# Формат вывода аудита

Получив ответ модели-аудитора от `hermes -z`, не пересказывай его «как есть» —
структурируй и сгруппируй по критичности. Перепроверь находки своими
инструментами (`Read`/`Grep`/`Bash`), прежде чем подавать как факт: модель-аудитор
тоже может ошибаться или галлюцинировать.

## Шаблон отчёта

```markdown
## Результаты аудита: [что аудировали]

**Модель-аудитор:** [например, DeepSeek 4 Pro @ openrouter]
**Линза:** [Скептик / Безопасность / …]
**Объём:** [строк / файлов / коммитов]

### 🔴 Critical
- [проблема] — [файл:строка] — [почему критично] — [предлагаемый фикс]

### 🟠 High
- …

### 🟡 Medium
- …

### 🟢 Low
- …

### Статистика
Всего: X · Critical: N · High: N · Medium: N · Low: N
```

Если модель вернула «Проблем не найдено» — так и сообщи, не выдумывай замечания.
Если проект без harness-документации (Шаг 0) — добавь дисклеймер в начало.

## Severity — ориентиры

- **Critical** — приводит к падению, потере/утечке данных, RCE, явно неверной
  бизнес-логике. Чинить до коммита/деплоя.
- **High** — серьёзный баг или уязвимость на реальном пути выполнения, но не
  мгновенно разрушительный.
- **Medium** — проблема надёжности/производительности на крайних случаях,
  заметный техдолг.
- **Low** — стиль, читаемость, мелкие улучшения.

## После отчёта — предложи

- **Исправить** найденное — сам (Claude Code) или вернуть задачу с замечаниями.
  Приоритет: Critical → High.
- **Перепроверить Critical другой моделью** — повторный `hermes -z` с иной
  моделью/линзой для подтверждения самых серьёзных находок (снижает шанс
  ложного срабатывания аудитора).
- **Сохранить отчёт** в заметки проекта (если пользователь ведёт Obsidian — см.
  навык `obsidian`).
