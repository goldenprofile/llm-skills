# Промты аудитора

Эти промты отправляются модели-аудитору через `hermes -z "..."`. Базовый промт —
основа; промт-добавку под линзу (см. [lenses.md](lenses.md)) вставляй сразу после
него, перед кодом.

## Базовый промт

```
Ты — senior-разработчик, делающий code review. Твоя задача — найти проблемы в
коде ниже. Код написан Claude Code (AI-ассистентом). Claude Code склонен к
типичным ошибкам:

1. Галлюцинации: вызов методов/API/полей, которых не существует.
2. Пропущенные импорты.
3. Over-engineering — абстракции ради абстракций.
4. Отсутствие обработки ошибок и граничных случаев.
5. Несоответствие существующим паттернам проекта.
6. Проблемы с типами (type mismatches).
7. Race conditions и проблемы конкурентности.

[СЮДА — промт-добавка линзы, если она задана]

[СЮДА — КОНТЕКСТ ПРОЕКТА: ключевые правила из CLAUDE.md/ARCHITECTURE.md]

[СЮДА — КОД НА ПРОВЕРКУ]

Выдай ОТСОРТИРОВАННЫЙ по критичности список проблем. Для каждой:
- Серьёзность: critical / high / medium / low
- Конкретное место (файл:строка или фрагмент)
- В чём проблема
- Предлагаемое исправление

НЕ хвали код. НЕ пиши «в целом хорошо». Только проблемы. Если проблем нет —
ответь ровно «Проблем не найдено» (это нормальный результат).
```

## Сборка промта (Claude Code)

1. Возьми базовый промт.
2. Если задана линза — вставь её промт-добавку из [lenses.md](lenses.md).
3. Вставь **выжимку правил проекта** из `CLAUDE.md`/`ARCHITECTURE.md` (Шаг 0),
   особенно запреты и правила сохранения моделей — иначе аудитор предложит то,
   что проект явно запрещает.
4. Вставь код (diff, содержимое файлов или описание архитектуры).
5. Передай всё одной строкой в `hermes -z "..."`. Для многострочного промта в
   PowerShell используй here-string, в bash — heredoc, либо запиши промт во
   временный файл и передай его содержимое.

## Промт по режимам

**Diff-аудит** — добавь к базовому:

```
Это git diff. Анализируй ТОЛЬКО изменённые строки (контекст вокруг — для
понимания). Не комментируй неизменённый код, кроме случаев, когда изменение
ломает его.
```

**Файловый аудит** — добавь:

```
Ниже — полные файлы. Оцени их целиком: корректность, связность, дублирование,
соответствие паттернам проекта.
```

**Архитектурный аудит** — добавь:

```
Оцени АРХИТЕКТУРУ, а не только построчную корректность:
- Связность и зависимости между модулями, нарушения слоёв.
- Принципы SOLID, избыточные/недостающие абстракции.
- Не изобретён ли велосипед там, где есть стандартное решение фреймворка/языка?
- Масштабируемость и точки расширения.
```

## Большой объём

Если diff/файлы >~500 строк — не отправляй одним запросом: модель потеряет
контекст. Дроби по файлам или логическим модулям, делай отдельный вызов
`hermes -z` на каждую порцию, затем объедини находки в общий отчёт. Для крупных
объёмов выбирай модель с большим контекстным окном (DeepSeek Pro, Kimi).
