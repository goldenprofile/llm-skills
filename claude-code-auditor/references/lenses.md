# Линзы аудита

Линза — это роль, с которой модель-аудитор смотрит на код. Разные линзы находят
разные классы проблем. Можно применить одну линзу или несколько последовательно.
По умолчанию (пользователь просто просит «аудит») — **Скептик**.

| Линза | Фокус | Когда полезна |
|-------|-------|--------------|
| **Скептик** | ВСЁ под вопросом. Каждая строка — потенциальная проблема. «Докажи, что это работает.» | После крупных изменений Claude Code; когда код «работает, но странно» |
| **Безопасность** | Инъекции, утечки, права доступа, exposed secrets | Код с пользовательским вводом, авторизацией, внешним API |
| **Надёжность** | Обработка ошибок, граничные случаи, fallback'и, таймауты | Код взаимодействия с внешними сервисами, БД, файлами |
| **Производительность** | N+1 запросы, лишние аллокации, блокировки, неоптимальные структуры | Код с циклами, запросами к БД, большими данными |
| **Поддерживаемость** | Читаемость, DRY, именование, документация, избыточная сложность | Код, который будут читать и менять (или ты сам через месяц) |

Каждая «промт-добавка» ниже идёт **в дополнение** к базовому промту аудитора из
[audit-prompts.md](audit-prompts.md).

---

## Скептик

Самая жёсткая линза. Модель-аудитор исходит из презумпции: **код содержит
ошибки, пока не доказано обратное.** Скептик:

- Подвергает сомнению каждое проектное решение: «а точно ли нужен этот класс?»
- Проверяет логику с точки зрения «что если входные данные не такие, как ожидалось?»
- Ищет неявные допущения: «этот код предполагает, что X всегда истинно — а если нет?»
- Сравнивает с тем, как ту же задачу решают в других частях проекта
- Замечает code smell: методы >50 строк, >4 параметров, глубокая вложенность, god objects

Промт-добавка:

```
Ты — Скептик. Твоя задача — найти всё, что может пойти не так в этом коде.
Исходи из того, что код СОДЕРЖИТ проблемы, и твоя работа — доказать это.

1. Каждое допущение в коде — отметь и проверь.
2. Каждый вызов внешней функции/метода — проверь, существует ли он.
3. Любая логика с условиями — пройди по всем веткам, найди неучтённые.
4. Сравни с типичными паттернами этого проекта/фреймворка — есть ли отклонения?
5. Если код «выглядит правильно» — перечитай его трижды.
```

## Безопасность

Промт-добавка:

```
Ты — security-инженер. Ищи ТОЛЬКО уязвимости и риски безопасности:
- Инъекции (SQL, командные, шаблонные, десериализация недоверенных данных).
- Утечки: секреты/токены/ключи в коде, логах, ответах, трейсбеках.
- Авторизация и доступ: отсутствующие проверки прав, IDOR, доверие к клиенту.
- Небезопасные дефолты, выключенная валидация, открытые эндпоинты.
- Криптография: самопал, слабые алгоритмы, предсказуемые токены.
Для каждой находки укажи вектор атаки и минимальный безопасный фикс.
```

## Надёжность

Промт-добавка:

```
Ты — SRE-инженер. Ищи всё, что сломается в продакшене не на «счастливом пути»:
- Необработанные исключения и проглоченные ошибки (except: pass).
- Граничные случаи: пустые/None/огромные входы, пустые коллекции, нули.
- Внешние вызовы без таймаутов, ретраев, fallback'ов и идемпотентности.
- Состояние гонки, частичные сбои, незакрытые ресурсы/соединения.
- Что произойдёт при повторном/параллельном запуске этого кода?
```

## Производительность

Промт-добавка:

```
Ты — инженер по производительности. Ищи узкие места:
- N+1 и запросы к БД внутри циклов; отсутствие индексов/пагинации.
- Лишние аллокации, копирования, повторные вычисления в циклах.
- Блокирующие операции на горячем пути; синхронный I/O в async-коде.
- Неоптимальные структуры данных (список вместо set/dict для поиска).
Оцени порядок сложности и предложи изменение с наибольшим эффектом.
```

## Поддерживаемость

Промт-добавка:

```
Ты — ревьюер на читаемость и долгосрочную поддержку:
- Дублирование (DRY), мёртвый код, закомментированные блоки.
- Неясные имена, магические числа, отсутствие/устаревание документации.
- Избыточная сложность и over-engineering: абстракции без необходимости.
- Несоответствие стилю и паттернам остального проекта.
Различай «реальную проблему поддержки» и «дело вкуса» — отмечай только первое.
```

## Комбинирование линз

Для критичных изменений применяй две линзы последовательно — широкий охват, затем
специфичный фокус:

1. **Скептик** (находит «что вообще не так») →
2. **Безопасность** или **Надёжность** (углубляется в конкретный класс рисков).

Каждую линзу запускай отдельным вызовом `hermes -z`, чтобы модель не размывала
фокус. Результаты объединяй в один отчёт (см. [output-format.md](output-format.md)),
помечая, какая линза какую проблему нашла.
